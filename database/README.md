# Database — Drizzle ORM

Source of truth for the AAPel PostgreSQL schema.

See [ARCHITECTURE.md §5](../ARCHITECTURE.md#5-modelo-de-dados) for the full data model rationale.

## Layout

```
database/
├── schema.ts                          # Drizzle schema (source of truth)
├── client.ts                          # Drizzle client for BFF/seed scripts
├── seed.ts                            # Initial seed data (run via pnpm db:seed)
├── migrations/                        # Auto-generated by drizzle-kit
│   └── _custom/
│       └── 0001_food_only_trigger.sql # Triggers (cross-table CHECK)
└── README.md
```

## Workflow

```bash
# 1. Edit schema.ts

# 2. Generate migration (creates SQL in ./migrations/)
pnpm db:generate

# 3. Apply to local DB
pnpm db:migrate

# 4. Apply custom triggers (after first migration)
psql $DATABASE_URL -f database/migrations/_custom/0001_food_only_trigger.sql

# 5. Seed initial data
pnpm db:seed

# Inspect interactively
pnpm db:studio
```

## Trade-offs

- **`pnpm db:push`** is faster for prototyping (no migration files) but should
  not be used after the first deploy. Use `db:generate` + `db:migrate` once the
  schema stabilizes.
- **Cross-table CHECK constraints** (e.g. "basket option must reference a FOOD
  product") cannot be expressed in Postgres CHECK syntax — they live in
  `_custom/` as triggers. This is documented in `schema.ts`.
- **Decimal columns are returned as `string`** by Drizzle (numeric precision).
  The BFF/FastAPI layer is responsible for parsing.

## Database setup (local Docker)

```bash
docker run -d \
  --name aapel-db \
  -e POSTGRES_USER=aapel \
  -e POSTGRES_PASSWORD=aapel \
  -e POSTGRES_DB=aapel \
  -p 5432:5432 \
  postgres:16
```

Then `cp .env.example .env` and `pnpm db:push` to bootstrap.
